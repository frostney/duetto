# Deployment

## Executive Summary

- duetto is a **library** — releases are git tags consumed via lwpt's dependency mechanism, not shipped binaries.
- Versions follow SemVer; tags are `<major>.<minor>.<patch>` (no `v` prefix — the convention across frostney Pascal projects, matching lwpt and GocciaScript) matching `version` in `lwpt.toml`.
- `CHANGELOG.md` is generated by git-cliff from Conventional Commits; release notes are never hand-edited.
- `ci.yml` on main is the release gate: platform matrix + Autobahn suite must be green before tagging.

## Release flow

1. Confirm main is green in
   [ci.yml](../.github/workflows/ci.yml) — including the Autobahn job.
2. Bump `version` in `lwpt.toml` (SemVer) on a PR; squash-merge it.
3. Tag the merge commit `<version>` (unprefixed) and push the tag.
4. Regenerate the changelog: `git-cliff -o CHANGELOG.md` and land it via PR.

If wording in the changelog is wrong, fix the underlying commit message
convention going forward and regenerate — entries are not hand-edited.

## Consumption

Downstream lwpt projects depend on duetto in their `lwpt.toml`:

```toml
[dependencies]
duetto = "frostney/duetto@^0.1.0"
```

During the sandbox phase the dependency can also be a local path
(`duetto = "../duetto"`), mirroring how duetto itself consumes lwpt's packages.
